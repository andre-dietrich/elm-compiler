# Kernel-List shape padding тАФ design

Date: 2026-07-18

## Context

`b8befd98` already gives every **Elm-defined** ADT union a monomorphic
object shape in `Mode.Prod` (all variants null-padded to the union's max
arity), so V8 sees one hidden class per union instead of one per variant.
That change explicitly excluded Kernel-defined types тАФ `List`'s `Cons`/`Nil`
are hand-written in `elm/core`'s `List.js` kernel file, not generated by
`Optimize.Module.addUnion`/`Generate.JavaScript.Expression.generateCtor`, so
they never went through the padding logic at all:

```js
var _List_Nil  = { $: 0 };
// ... _List_Cons(hd, tl) builds { $: 1, a: hd, b: tl }
```

Every list pattern-match hot path (`!list.b`) toggles between these two
different hidden classes each time it crosses a `Cons -> Nil` boundary.

A hand-patch spike (2026-07-17, see memory `kernel-list-padding-spike`)
confirmed the effect is real but **conditional on code shape**: padding
`_List_Nil` to `{ $: 0, a: null, b: null }` (matching `Cons`'s field order)
gave **+8.1тАУ8.5%** for code that builds many short lists (frequent
`Cons`<->`Nil` shape transitions), and a neutral **~0%** (never negative)
for a single long list traversed once (the transition happens once,
amortized away). This design turns that validated hand-patch into a real
compiler change.

## Key finding that simplifies this vs. the original plan sketch

The original spike write-up assumed this would require patching `elm/core`
itself (a separate package, outside this repo) тАФ a materially bigger and
riskier undertaking than every other perf change in this fork, which only
ever touch this compiler's own codegen.

That's not necessary. `AST.Optimized.addKernel` (`AST/Optimized.hs:236-245`)
bundles an **entire** kernel `.js` file (e.g. all of `List.js` тАФ `_List_Nil`,
`_List_Cons`, `cons`, everything else it defines) into a single
`Opt.Kernel [K.Chunk] deps` graph node, keyed by one synthetic global:

```haskell
toKernelGlobal :: Name.Name -> Global
toKernelGlobal shortName =
  Global (ModuleName.Canonical Pkg.kernel shortName) Name.dollar
```

Whole-program DCE operates at the granularity of this single node тАФ if
*anything* from `List.js` is reachable, the whole file's text (including the
`_List_Nil` declaration) is emitted verbatim by
`Generate.JavaScript.generateKernel` (`Generate/JavaScript.hs:357-359`).
That means we can append one extra, unconditionally-safe statement right
after this node's own text, gated only on "is this the `List` kernel node"
and "are we in Prod mode" тАФ entirely inside `Generate/JavaScript.hs`, no
`elm/core` fork, no new `AST.Optimized` constructor, no `.elmo` binary
format change (unlike `b8befd98`/TRMC, which both had to bump `Data.Binary`
tags and invalidate every cached `.elmo`/package cache).

## Mechanism

In `Generate/JavaScript.hs`'s `addGlobalHelp`, the `Opt.Kernel` branch
(currently `Generate/JavaScript.hs:222-226`):

```haskell
Opt.Kernel chunks deps ->
  if isDebugger global && not (Mode.isDebug mode) then
    state
  else
    addKernel (addDeps deps state) (generateKernel mode chunks <> padListNil mode global)
```

New helper, next to `isDebugger`:

```haskell
padListNil :: Mode.Mode -> Opt.Global -> B.Builder
padListNil mode global =
  case mode of
    Mode.Prod _ _ | global == Opt.toKernelGlobal "List" ->
      "_List_Nil = { $: 0, a: null, b: null };"
    _ ->
      mempty
```

`Opt.toKernelGlobal` is already exported from `AST.Optimized` and is the
exact same function `addKernel` used to build this node's key in the first
place, so the comparison can't drift from how the graph actually keys the
`List` kernel file. `OverloadedStrings` is already enabled in this module
(used for `"_UNUSED"` on the `Debug`/`Prod` chunk branches directly above),
so both the `Name.Name` literal `"List"` and the `B.Builder` literal both
type-check with no new imports.

The appended statement is a full-literal **reassignment**, not an
incremental `.a =`/`.b =` mutation тАФ this reproduces byte-for-byte the
object-shape-construction code path already benchmarked in the spike (a
single object literal with all three fields present from the start), so no
new hidden-class-transition behavior is introduced that the spike didn't
already measure. `_List_Nil` is declared with `var` earlier in the same
concatenated kernel text, so a plain (non-`var`) reassignment is valid JS
and executes immediately after, before any user/program code runs (all
kernel code runs at module load, ahead of `main`).

## Scope / safety

- **Mode-gated**: only fires under `Mode.Prod`; `Mode.Dev` output is
  untouched (`padListNil` returns `mempty` for `Mode.Dev`), preserving the
  CLAUDE.md contract that Prod-only codegen must never change Dev output.
- **No `.elmo`/binary format change** тАФ pure `Generate`-time text
  concatenation. No cache invalidation needed for this change (contrast
  with `b8befd98`/TRMC).
- **No `elm/core` changes** тАФ the padding is injected by this compiler,
  independent of the exact literal source text of `List.js` in whatever
  version of `elm/core` a project uses.
- **Correctness** (carried over from the spike's pre-verified analysis):
  `_Utils_eqHelp`'s `for (var key in x)` structural-equality walk still
  works тАФ `Nil`-vs-`Nil` compares the new `null` fields trivially equal,
  `Nil`-vs-`Cons` still short-circuits on the `$` tag mismatch first.
  `!list.b`-style pattern-match checks on `Nil` are unaffected (`null` stays
  falsy). `Dict`/`Array`/`Set` and any other kernel/library code that
  builds or consumes `List` values internally goes through the same
  `_List_Nil` singleton, so they're covered automatically тАФ nothing else
  needs touching.
- **Out of scope**: this does not address `Debug.toString`/debugger display
  of lists (Dev mode is untouched, so no impact); does not touch any other
  Kernel-defined type (only `List` was validated by the spike).

## Testing / verification

No automated suite exists in this repo (per `CLAUDE.md`); manual protocol,
consistent with every prior perf commit in this fork:

1. Build the compiler in the Docker toolchain (`-O0` for the iterate loop).
2. Compile a trivial scratch module that uses `List` in `--optimize` mode;
   diff the generated JS against a pre-change build. Expect **exactly one**
   added line (the `_List_Nil = {...}` reassignment), nothing else changed.
3. Compile the same module without `--optimize` (Dev); diff against a
   pre-change Dev build тАФ expect **zero** difference.
4. Re-run the two benchmarks from the spike (`sumShortLists`: 5,000,000├Ч
   fresh 5-element list via `List.range 1 5` + `List.sum`; `sumOneLongList`:
   one-shot `List.sum (List.range 1 5000000)`), interleaved process runs,
   checksums verified identical each run. Expect: short-lists case ~8%
   faster (matching the spike), long-list case ~0% (neither faster nor
   slower, within noise) тАФ confirming the real compiler's output behaves
   the same as the spike's hand patch.
5. Sanity-check a small program that uses `Dict`/`Array`/`Set` (which build
   `List` values internally, e.g. `Dict.toList`) under Node to confirm no
   runtime regression from the padded `Nil` singleton.
