# Benchmark Showcase — Design

Date: 2026-07-21

## Purpose

`benchmark/` (currently empty) becomes a static, browser-viewable showcase that demonstrates the
cumulative effect of this fork's JS-codegen optimizations by compiling the *same* small Elm programs
with the official Elm compiler and with this fork's patched compiler, running both side by side, and
letting the numbers speak. It also demonstrates the new `Worker.run` feature (Web Worker offloading),
which has no equivalent in the official compiler at all.

This is a demo/communication artifact, not a regression-test suite. Exact speedup numbers will vary by
machine; the numbers quoted below (from prior spec/memory files) are expectations to sanity-check
against, not contractual thresholds.

## Non-goals

- No automated pass/fail assertions on the measured numbers — a human looks at the page and compares.
- No live/on-demand recompilation in the browser — everything is prebuilt to static JS.
- No attempt to make the two Worker.run demo variants share one Elm source file — see "Worker section"
  below for why that's structurally impossible here.

## Baseline compiler & build pipeline

- **Old**: the official Elm compiler, already installed locally at `/usr/local/bin/elm`, reports itself
  as `0.19.1`.
- **New**: `./elm-patched` at the repo root (this fork, built via the Docker recipe in `CLAUDE.md`),
  reports itself as `0.19.2`. Verified current (contains the `Worker.run` error strings from the most
  recent commit) via `strings elm-patched | grep -i "Worker.run"`.
- **Verified cache-safety fact**: `builder/src/Stuff.hs:39` computes both the project-local
  `elm-stuff/<compiler-version>/` cache dir and the `ELM_HOME/<compiler-version>/` package cache using
  the compiler's own reported version string. Because the two compilers report different versions
  (`0.19.1` vs `0.19.2`), their caches never collide even when pointed at the same project directory and
  the same `ELM_HOME` (`~/.elm`). The "stale cache" gotcha recorded in prior memory notes only applies
  when comparing two builds that report the *same* version (e.g. two fork commits) — not relevant here.
  This means `build.sh` can compile each example twice, back to back, in the same project dir, with no
  special cache isolation.
- `build.sh` follows the existing `sourcemap-demo/build.sh` pattern (rebuild `elm-patched` via the
  `haskell:9.8.4` Docker image only if stale; otherwise reuse the binary already at the repo root) and,
  for each example directory, runs:
  ```bash
  elm make src/Main.elm --optimize --output=dist/old.js       # official compiler, on PATH
  ./elm-patched make src/Main.elm --optimize --output=dist/new.js
  ```
  The repo's existing `.gitignore` already excludes any `dist` directory, so `dist/old.js` / `dist/new.js`
  per example are build artifacts regenerated by `build.sh`, not committed — consistent with how
  `elm-stuff` is already handled.
- A local static file server (`python3 -m http.server`, matching `sourcemap-demo`'s documented workflow)
  serves `benchmark/` for viewing. No other server/backend involved.

## Directory structure

```
benchmark/
  build.sh                        # orchestrates all per-example compiles (both compilers)
  index.html                      # the single showcase page
  shared/
    style.css                     # shared layout/visual styles
    harness.js                    # lazy-iframe-start helper + window.onerror banner (see below)
  examples/
    list-pipeline-fusion/
      elm.json
      src/Main.elm                # SAME source compiled by both compilers
      dist/old.js  dist/new.js
      frame-old.html  frame-new.html   # iframe content: loads dist/old.js or dist/new.js, mounts Benchmark.Runner
    html-producer-chain-fusion/    (same shape)
    trmc-stack-safety/             (same shape, custom Elm app instead of Benchmark.Runner)
    kernel-list-shape-padding/     (same shape)
    unwrapped-hofs/                (same shape)
    record-update-inlining/
      elm.json
      src/MainOld.elm             # DIFFERENT source — see below
      src/MainNew.elm
      dist/old.js  dist/new.js
      frame-old.html  frame-new.html
    worker-fibonacci/
      elm.json                    # new/ elm.json depends on andre-dietrich/worker; old/ does not
      src/MainOld.elm             # plain blocking recursion, no Worker import
      src/MainNew.elm             # uses Worker.run
      dist/old.js  dist/new.js
      frame-old.html  frame-new.html
    worker-ackermann/              (same shape as worker-fibonacci)
```

Every example uses the same `frame-old.html` / `frame-new.html` pairing for consistency, even where the
two sides share one `src/Main.elm` — the only difference between the pair is which compiled script tag
they load (`dist/old.js` vs `dist/new.js`). Only the two special-cased examples (record-update-inlining,
and both Worker examples) additionally need genuinely different *Elm source* per side, as explained
per-section below.

## Shared harness (`shared/harness.js`)

Two small, generic pieces used by every example:

1. **Lazy iframe start.** Each example's iframe starts with no `src` attribute. Its "▶ Start" button
   sets `iframe.src = "examples/<name>/frame-<variant>.html"` on click. This is what gives every
   example/compiler pair its own independent, on-demand start — nothing runs until clicked, so the page
   never runs six-plus heavy computations at once. (For `Benchmark.Runner.program`, which has no built-in
   manual start/stop and begins sampling as soon as it loads, this lazy-`src` trick *is* the start
   button — no custom pause/resume logic needed inside Elm.)
2. **Visible crash banner.** A `window.addEventListener('error', ...)` /
   `window.addEventListener('unhandledrejection', ...)` handler installed in each `frame*.html` (via a
   shared inline snippet) that renders a red on-page banner with the error message. This exists because
   Elm's own runtime cannot catch a JS-level `RangeError: Maximum call stack size exceeded` from pure Elm
   code — without this, the TRMC stack-safety demo's "old" side would only show the crash in the devtools
   console, not on the page.

## The 6 comparison examples

### 1. List Pipeline Fusion
`Benchmark.describe` with several pipeline depths over `List.range 1 200000`:
`filter → map → filter → foldl` (2/3/4-stage) and `filter → map → sum`. Same source, both compilers.
Expected: 8×–16× (from `list-foldl-fusion-plan.md` / `list-fusion-v2-spike.md`). The headline example —
put it first.

### 2. Html/Producer-Chain Fusion
`filter → map → filter → map` chain with no fold at the end (the shape of `Html.ul [] (List.map view
(List.filter ok items))`), benchmarked as a pure function returning a `List`. Same source, both
compilers. Expected: 2.75×–8.17× (`bare-producer-chain-fusion-spike.md`).

### 3. TRMC Stack-Safety
Not a throughput benchmark — a "does it crash" demo. Recursively builds a list/tree of length
N=1,000,000. Custom small Elm app (Start button, no `elm-explorations/benchmark`) showing either
"✅ completed in Xms" or (via the shared crash banner) "💥 RangeError: Maximum call stack size exceeded".
Same source, both compilers. Old is expected to crash around N≈6,300–6,700 (per `trmc-plan.md`); new
completes in ~90–100ms.

### 4. Kernel List Shape Padding
Many short, freshly-constructed lists summed repeatedly (`List.range 1 5 |> List.sum`, looped). Same
source, both compilers. Expected: smallest number in the set, ~+8% (`kernel-list-shape-padding-plan.md`).
Deliberately included as the "not every win is a headline number" data point.

### 5. Unwrapped HOFs
`map`/`foldr` pipeline over a large list. Same source, both compilers. Expected: +27–28%
(`unwrapped-hofs-plan.md`).

### 6. Record-Update Inlining (dotted-path syntax)
Structurally different from the five above: this fork added new surface syntax (dotted paths inside
`{ … | … }`, plus `<-`/`.{}` forms — commits `009a3694`, `4717e5ae`, `1b83f52c`, all already on `main`)
that the official compiler cannot parse at all. So old and new run genuinely different, hand-verified
*semantically equivalent* source:
- **New** (`src/MainNew.elm`): `{ model | account.address.city = "Berlin" }` — one line, relies on
  Tier2's always-inline Prod codegen (`be3dd455`).
- **Old** (`src/MainOld.elm`): the verbose hand-nested equivalent Elm developers had to write before —
  `{ model | account = { account0 | address = { address0 | city = "Berlin" } } }` — compiled by the
  official compiler, which has none of this fork's inlining.

`Benchmark.describe` with a couple of record sizes/nesting depths to show the effect scaling with field
count. Expected: +7–35% (`tier2-record-update-spike.md`), plus the syntax/ergonomics contrast itself is
worth calling out in the example's description text.

## Worker section (below the 6 examples, own heading)

Uses `Worker.run : (a -> b) -> a -> Task x b` from `andre-dietrich/worker` (installed locally at
`~/.elm/0.19.2/packages/andre-dietrich/worker/1.0.0`; exposes only `Worker.run`/`Worker.batch`). Verified
directly from that package's `src/Worker.elm` and `src/Elm/Kernel/Worker.js`:

- `fn` must be a bare top-level unary function with port-safe argument/result types (same allow-list as
  ports: numbers, bools, strings, `Maybe`/`List`/`Array`, tuples, closed records, JSON values).
- The worker-side dispatcher wraps the call in `try/catch` and replies with `{ error: String(e) }` on any
  exception — so a stack overflow *inside* the worker surfaces as a clean `Task` failure
  (`Err "RangeError: ..."`), not a crash, and the main thread is never blocked while it happens.
- The official compiler doesn't trust `andre-dietrich` as a kernel-code author (`Elm.Package.isKernel` is
  fork-specific), so it would flatly refuse to compile anything that imports `Worker`. **The "old" side of
  both worker examples is therefore plain, `Worker`-free, blocking Elm code** — not a compile-error
  showcase, just the same computation run synchronously on the main thread. This mirrors the
  record-update example's old/new-are-different-source structure.
- The demo needs a real browser (not Node) to show real offloading — the kernel code has an explicit
  `typeof Worker === 'undefined'` fallback that runs synchronously in-process, which is correct behavior
  for non-browser contexts but wouldn't demonstrate anything here.

### Fibonacci (responsiveness demo)
Naive exponential `fib n`; user enters `n` (suggest default ~35–38, tunable). Both variants show a
ticking counter/spinner driven by a `Time.every` subscription. Old: counter visibly freezes for the
duration of the blocking call. New: `Worker.run fib n` returns a `Task`, dispatched via `Task.attempt`;
counter keeps ticking throughout.

### Ackermann (graceful-failure demo)
Naive recursive `ackermann m n`; user enters `n` via a bounded input/slider (suggest `m=3` fixed,
`n` up to ~10–12). Old: blocks, then either finishes (small n) or throws an uncaught `RangeError` that the
shared crash banner surfaces, with the page having been unresponsive throughout. New: `Worker.run`
version's `Task` either succeeds or resolves to a clean `Err "RangeError: ..."` shown inline in the Elm
UI (not just the crash banner) — no freeze at any point. **Exact `n` thresholds for a reliable crash need
empirical calibration during implementation** (Ackermann's recursion depth is not a simple function of
its inputs) — pick values that crash reliably within a few seconds, not ones that hang for minutes first.

## Page layout

Single `benchmark/index.html`. One `<section id="...">` per example plus one for the Worker demos (with
its own two subsections), in the order listed above, each with an anchor-linkable heading, a short
description quoting the expected numbers as a teaser, a `<pre><code>` block with the representative Elm
snippet (plain text, no syntax-highlighting dependency), and two columns ("Alt (Elm 0.19.1)" /
"Neu (dieser Fork)") each with its own "▶ Start" button and lazily-sourced iframe. No cross-frame
`postMessage` aggregation in v1 — the user visually compares the two panels; each `Benchmark.Runner` /
custom result view reports its own number independently.

## Testing / verification plan

Manual, matching this repo's existing "no automated test suite" convention (per `CLAUDE.md`):

1. Run `benchmark/build.sh`; confirm both `dist/old.js` and `dist/new.js` exist for every example with no
   compile errors (the record-update and worker "old" variants are expected to compile cleanly — they
   deliberately don't use the new features, not attempts to fail).
2. Serve `benchmark/` via `python3 -m http.server` and open it in a real browser (needed for `Worker.run`
   to actually offload — see above).
3. Click through every Start button once per side; confirm:
   - The 5 `Benchmark.Runner`-based examples render ops/sec numbers that stabilize, with the new side
     visibly ahead of the old side by roughly the magnitude noted per example above.
   - The TRMC example's old side shows the crash banner; new side shows a completion time.
   - The record-update example's two sides show different source in their code snippets but the same
     described transformation, with new visibly faster.
   - Fibonacci: old side's counter freezes during computation; new side's counter keeps ticking.
   - Ackermann: old side eventually shows the crash banner after an unresponsive period; new side shows
     an inline `Err "RangeError..."` (or success, for small n) with no freeze.
4. Spot-check that clicking Start on one example while another is mid-run doesn't interfere (they're
   independent iframes/documents, so this should hold structurally, but verify once in practice).
